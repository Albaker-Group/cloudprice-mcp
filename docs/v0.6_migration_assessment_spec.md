## Goal

Add an **executive-grade migration assessment** as one MCP tool + one CLI subcommand. Given a workload portfolio currently running on cloud X, project the steady-state monthly cost on Y/Z/W, the one-time exit cost (egress to leave X), and the payback period — with a small caveats library that flags known cross-cloud risk (ARM, region gaps, service substitutions).

This is **assessment-only**. Not actual VM transfer, not IaC translation, not network rewriting. Those are a separate product.

## Why this fits cloudprice-mcp

The v0.5 catalog already has every input we need: compute SKUs, block storage, object storage, managed Postgres, egress (internet + inter-region). What's missing is the **integration layer** that:

1. Runs every comparator under the hood for a single portfolio.
2. Aggregates totals per target cloud.
3. Adds one-time exit cost (the number nobody else surfaces cleanly).
4. Computes payback and 3-year TCO.
5. Returns a ranked recommendation with caveats.

Zero new pricing data. Pure orchestration.

## User-facing surface

### MCP tool

`assess_migration` — single tool, accepts an inventory in any of three formats:

```json
{
  "inventory_format": "yaml" | "csv" | "json",
  "inventory_content": "<raw string>",
  "export_format": "markdown" | "csv" | "json" | "text" | "html",
  "targets": ["azure", "gcp", "oci"],   // optional; default = all clouds except source
  "commitment": "1yr_no_upfront",
  "multi_az": true
}
```

Or, for clients that already have a parsed object:

```json
{
  "inventory": { ... structured object ... },
  "export_format": "markdown"
}
```

### CLI subcommand

```
cloudprice-mcp assess <file> [--target oci] [--export markdown|csv|json|text|html]
cloudprice-mcp assess --template yaml > workloads.yaml
cloudprice-mcp assess --template csv  > workloads.csv
cloudprice-mcp assess --schema
```

The CLI works on a VM with no AI — doubles as content marketing ("one command, see your savings").

## Inventory schema (canonical, format-independent)

```yaml
source_cloud: aws            # required — the cloud we're costing AGAINST
commitment: 1yr_no_upfront   # optional — applies to TARGET compute only
multi_az: true               # optional — applies to TARGET compute only
one_time:
  data_to_migrate_gb: 50000  # required if exit cost should appear

compute:
  - { name, vcpus, memory_gb, quantity, multi_az?, os_disk_gb?, os_disk_type?, snapshot_count?, snapshot_incremental_factor? }

storage:
  - { name, disk_type, capacity_gb, quantity?, snapshot_count?, snapshot_incremental_factor? }

object_storage:
  - { name, capacity_gb, tier, quantity? }      # tier ∈ hot|cool|archive

databases:
  - { name, engine, vcpus, memory_gb, storage_gb, quantity? }   # engine = postgres only in v0.6

egress:
  - { name, gb_per_month }                       # internet egress
  - { name, gb_per_month, kind: inter_region }   # cross-region within same target cloud
```

### CSV (wide format)

Type discriminator in column 1, all attributes as columns:

```csv

compute,api,4,16,6,,,,,,,,,
compute,worker,8,32,3,,,,,,,,,

db,primary,8,32,,,,,,postgres,500,,
egress,internet,,,,,,,,,,5000,
```

Header row also accepts `source_cloud=aws` and similar key=value rows in column 1 for top-level fields.

### YAML / JSON

Same shape as the canonical schema above. YAML is the recommended default — readable, comment-friendly, matches what IaC users already write.

`export_format` controls the rendering of this same data:

| Format | Audience |
|---|---|
| `markdown` (default) | Slack / docs / LLM context — comparison table + headline numbers |
| `csv` | Excel / pivot — one row per target |
| `json` | Programmatic / piping |
| `text` | Plain ASCII for terminals or chat |
| `html` | Standalone HTML file with table for sharing |

## Caveats library (initial set, v0.6.0)

Stored in `data/caveats.json`. Each entry: `{cloud, applies_when, message, severity: info|warn|block}`.

| Cloud | Trigger | Message | Severity |
|---|---|---|---|
| oci | compute requested + A1 Flex selected | A1.Flex is ARM (Ampere) — x86 AMIs do not migrate without rebuild | warn |
| oci | gpu workload detected (future) | GPU SKUs limited to specific regions | warn |
| any | commitment=3yr_partial_upfront | RI/SP commitments are not portable; commit only after pilot | info |
| any | egress > 100 TB/month | High-volume egress benefits from direct connect / interconnect — not modeled | info |
| any | source_cloud == target | No-op — target equals source | block |

Library is intentionally small at launch. Grow it from real user issues, not our imagination.

## Honest gaps (we publish these in the output)

Every assessment response includes `honest_gaps`. This is a **trust-building feature**, not a disclaimer dump. It tells the executive reader:

1. We do not do workload discovery — they brought the inventory.
2. We do not model license portability (Windows BYOL, SQL Server, RHEL subs).
3. We do not right-size from CPU utilization (we trust the supplied vCPU/RAM).
4. We use list price + flat commitment tiers — actual rates depend on negotiation.
5. We do not model performance migration risk (only cost).

Without these, the tool reads like a marketing pitch. With them, it reads like a credible FinOps artifact.

- Multi-region target selection (v0.5.1 will add us-west / eu-west; assessment can use those once shipped).
- Engines other than Postgres (MySQL / SQL Server / Oracle DB) — wait for demand signal.
